About the Data

What is Argo?

Argo is an international scientific program that maintains ~4,000 autonomous robotic floats deployed across the world's oceans. Every 10 days each float dives to 2,000 metres, profiles the water column as it rises, then transmits data via satellite — giving scientists an unprecedented real-time picture of ocean health.

🤿

4,000+ Active Floats

Autonomous profiling floats deployed across every major ocean basin. Each float drifts with currents, dives to 2,000 m on a 10-day cycle, and surfaces to transmit.

🌡️

Multi-Parameter Sensing

Core floats measure temperature, salinity, and pressure. BGC-Argo floats add dissolved oxygen, chlorophyll-a, optical backscatter, and pH — critical for biological oceanography.

🛰️

Satellite Data Relay

After surfacing, floats transmit to global data centres — INCOIS, Coriolis, AOML, BODC — within hours. This project ingests INCOIS Argo data directly.

🌍

Global Scientific Impact

Argo data underpins IPCC climate assessments, hurricane intensity forecasting, fisheries management, sea-level studies, and carbon cycle research worldwide.

What This Project Offers

💬

Natural Language Querying

Ask "What is the average temperature at the equator?" — GPT-4.1-mini extracts your intent and the backend builds exact SQL automatically. No query syntax needed.

⚡

LRU Intent Caching

Repeated or similar questions skip the LLM entirely and return from an in-memory LRU cache. Cache hits answer in under 5 ms — zero API cost, zero latency.

🔗

Threaded Connection Pool

10 pre-warmed PostgreSQL connections reused across all requests. No TCP handshake per query — concurrent users are handled safely with thread locking.

Under the Hood

Optimised vs Unoptimised

Both panels return the same correct answer. The difference is every architectural decision made along the way.

🐌 Unoptimised  /chat

  • ✗ New OpenAI client object created on every single request
  • ✗ .env file re-read from disk inside every helper function
  • ✗ LLM called every time — no caching, identical questions hit the API again
  • ✗ Two separate DB connections opened per request (table-check + query)
  • ✗ Fetches ALL matching rows into Python RAM
  • ✗ AVG / MIN / MAX computed in a Python loop — not in the database
  • ✗ No SQL result cache — every identical query re-hits the database
Typical latency 3,000 – 8,000 ms
VS

⚡ Optimised  /chat_optimised

  • ✓ Single OpenAI client instantiated once at startup, reused forever
  • ✓ Config loaded once — no repeated disk reads
  • ✓ LRU intent cache (500 entries) — repeated questions skip the LLM entirely
  • ✓ ThreadedConnectionPool — connection borrowed in microseconds, not created
  • ✓ DB computes AVG()/MIN()/MAX() — one number sent back
  • ✓ LRU SQL cache (200 entries) — identical queries return instantly from memory
  • ✓ NULL filter on bio columns avoids scanning irrelevant rows
Typical latency 200 – 800 ms   (cache hit: <5 ms)
