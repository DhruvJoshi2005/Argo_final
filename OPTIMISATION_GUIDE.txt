Feature Additions

Shipped:
1. Depth range filter — fixed. depth_min/depth_max are extracted by the LLM (same
   pattern as lat_min/lat_max) and generate `pressure BETWEEN x AND y` instead of
   a broken exact match.
2. Vertical profile endpoint — GET /profile/{platform_number}/{cycle_number} returns
   pressure/temperature/salinity/doxy/chla/bbp700/ph ordered by pressure.
3. Float trajectory endpoint — GET /float/{platform_number}/track returns ordered
   (cycle, date, lat, lon) for map overlays.
7. CSV export endpoint — POST /export runs the same NL pipeline as /chat_optimised
   but streams raw rows as a CSV download (POST /export/count gives a size preview
   first).
8. /health endpoint — one DB query, returns float/cycle/measurement counts and the
   most recent observation timestamp.

Still open (next steps):
4. T-S (Temperature-Salinity) diagram data — a query mode returning paired
   (temperature, salinity, pressure) for a region instead of a single metric, so
   the frontend can plot a T-S scatter. Not yet built.
5. Time-series / trend queries — e.g. "how did temperature change over the months
   in the Bay of Bengal" needs a "trend" query mode grouping by
   date_trunc('month', juld). Not yet built.
6. Institution / float filter in chat — the flat table already has pi_name,
   project_name, institution, platform_number; the LLM intent extraction doesn't
   pull them yet (e.g. "temperature from INCOIS floats" → WHERE institution =
   'INCOIS'). Not yet built.

Placement Project Review
What's strong
End-to-end pipeline	Download → parse NetCDF → ingest → DB → API → NL chat. Very few placement projects have all five stages working.
Real dataset	16.8M rows of actual oceanographic sensor data. Not a toy CSV.
LLM integration	Intent → SQL pipeline with caching is genuinely well-designed. The LRU cache for both intents and SQL results shows you thought about cost.
Data engineering	QC filtering, NaN guard, position self-healing, correct array indexing — all fixed and documented. This is the kind of debugging that separates real engineers from tutorial-followers.
FastAPI with proper models	Pydantic response models, connection pooling, CORS — not just flask.jsonify.
